# 펫과나 개선 회의록

일시: 2026-06-18
참석: 예원(총괄), 영숙(운영), 코다리(개발), 케빈(인프라), 티모(디자인), 로율(법무), 현빈(전략), 아린(온보딩), 루나(콘텐츠), 데이브/레오(성장 실험)

## 회의 목적

펫과나를 기능 나열형 데모에서 실제 사용자가 반복해서 쓰는 반려동물 케어 서비스로 끌어올린다. 이번 회의는 새 기능을 더 붙이기보다, 현재 구현된 건강/산책/소셜/일기/결제/동기화 흐름 중 완성도가 낮은 병목을 정리하고 다음 실행 순서를 정하는 데 집중했다.

## 현재 확인된 상태

- `index.html`에 `GEMINI_API_KEY`, `STRIPE_PAYMENT_LINK`, `STRIPE_SHOP_PAYMENT_LINK`가 비어 있어 AI 건강 분석과 결제 전환이 실제 운영 흐름으로 닫히지 않는다.
- `README.md`와 `CHANGELOG.md` 기준 기능은 풍부하지만, `docs/progress.md`의 남은 작업은 여전히 결제 링크, webhook 검증, 소셜 Supabase 실시간 연동, 월별 건강 PDF다.
- 로그인/소셜/앨범/건강 데이터는 Supabase 함수가 붙어 있으나 localStorage fallback이 넓게 남아 있다. 데모 안정성은 좋지만 멀티 디바이스 신뢰성은 낮다.
- AI 건강 분석은 프론트에서 Gemini API를 직접 호출하는 구조라, 키를 넣는 순간 노출 위험이 있다. 서버 프록시 또는 Vercel API 라우트가 필요하다.
- UI는 기능 밀도가 높고 모바일 하단 탭 중심으로 구성되어 있다. 다음 개선은 큰 리디자인보다 핵심 플로우의 마찰 제거가 우선이다.

## 에이전트별 발언 요약

### 예원 - 제품 총괄

현재 제품은 기능 수가 강점이지만, 첫 사용자가 무엇을 먼저 해야 하는지가 흐릿하다. 첫 3분 경험을 `펫 등록 -> 건강 체크 -> 오늘 할 일`로 고정하고, 사주/소셜/쇼핑은 보조 흐름으로 뒤에 둔다.

결정: 다음 스프린트 목표는 "첫날 재방문 이유 만들기"다.

### 코다리 - 개발

가장 큰 기술 부채는 클라이언트 직접 API 키 사용과 localStorage fallback 난립이다. AI 건강 분석은 `/api/ai-health` 같은 Vercel Function으로 옮기고, Supabase 연결 실패 시 사용자에게 명확한 로컬 모드 표시를 해야 한다.

결정: AI 건강 분석 서버 프록시를 1순위로 만든다.

### 티모 - 디자인

기능이 많아 화면마다 카드와 버튼이 많다. 모바일에서는 "오늘의 케어" 한 화면에 건강 점수, 산책, 밥/물, 다음 일정만 먼저 보이게 하고, 상세 기능은 접히는 패널로 내려야 한다.

결정: 마이펫 첫 화면을 데일리 대시보드로 압축한다.

### 케빈 - 인프라

Vercel 환경변수와 실제 `index.html`의 빈 값이 맞물려 운영 기능이 비활성화되어 있다. 결제도 Payment Link 기반이라면 최소 성공/취소 URL과 premium 활성화 검증을 문서화해야 한다.

결정: 배포 전 환경변수 체크 스크립트와 Vercel 설정 점검표를 추가한다.

### 로율 - 법무

건강 분석은 수의학적 진단처럼 보이면 위험하다. 모든 AI 건강 결과에 "참고용이며 진단이 아님", "이상 증상은 병원 방문" 문구를 결과 카드 안에 넣어야 한다. 계정 삭제와 데이터 삭제 흐름도 실제 Supabase 삭제까지 확인해야 한다.

결정: AI 결과 카드와 탈퇴 흐름에 컴플라이언스 문구/검증을 추가한다.

### 영숙 - 운영

사용자 문의/CS 관점에서는 "왜 AI 분석이 안 되나요", "로그인했는데 데이터가 사라졌어요", "결제했는데 프리미엄이 안 돼요"가 제일 위험하다. 설정 탭에 연결 상태, 계정 동기화 상태, 프리미엄 상태를 보여줘야 한다.

결정: 설정 탭에 서비스 상태 패널을 추가한다.

### 현빈 - 전략

펫과나의 차별점은 기능 묶음이 아니라 "케어 루틴 데이터가 쌓이면 더 똑똑해지는 서비스"다. 월간 건강 리포트 PDF는 단순 PDF보다 retention 핵심 기능이다.

결정: 월간 리포트를 프리미엄 전환 핵심으로 둔다.

### 아린 - 온보딩

온보딩은 이미 있지만, 기능 설명 중심이다. 사용자가 첫 펫을 등록하지 않으면 이후 기능이 약해진다. 온보딩 완료 액션을 "첫 펫 등록"으로 강제 연결해야 한다.

결정: 온보딩 완료 후 빈 상태를 줄이고 펫 등록 CTA를 강화한다.

### 루나 - 콘텐츠

펫 감정 대사는 좋은 장치다. 하지만 건강/산책/밥 기록과 연결돼야 생동감이 생긴다. 기록을 하면 펫이 반응하고, 기록이 없으면 다정하게 유도하는 흐름을 강화한다.

결정: 데일리 케어 이벤트와 펫 감정 대사를 연결한다.

### 데이브/레오 - 성장 실험

결제 전환 전에 무료 사용자가 "이 앱이 내 펫을 기억한다"는 느낌을 받아야 한다. 무료 5회 AI 분석보다, 7일 건강 추세와 월간 리포트 미리보기가 더 강한 전환 장치다.

결정: 프리미엄 모달은 가격보다 저장된 리포트/추세의 가치를 먼저 보여준다.

## 우선순위 결정

### P0 - 운영 안전

1. AI 건강 분석을 Vercel API 프록시로 이전한다.
2. Gemini API 키를 프론트에서 직접 쓰지 않게 한다.
3. AI 분석 결과에 의료 고지 문구를 항상 표시한다.
4. 설정 탭에 Supabase/AI/결제 연결 상태를 표시한다.

### P1 - 핵심 사용 흐름

1. 마이펫 첫 화면을 `오늘의 케어 대시보드`로 재구성한다.
2. 펫 미등록/데이터 없음 상태의 CTA를 정리한다.
3. 건강 기록, 산책, 밥/물, 일정이 한 화면에서 이어지게 한다.
4. 온보딩 종료 후 첫 펫 등록으로 자연스럽게 이동시킨다.

### P2 - 동기화와 소셜 완성

1. localStorage-only 데이터를 Supabase 기준으로 정리한다.
2. 소셜 피드/친구 일기 공유를 실제 계정 기준으로 검증한다.
3. 이미지/앨범 미디어를 Supabase Storage에 안정적으로 저장한다.

### P3 - 수익화

1. Stripe Payment Link 환경변수를 채운다.
2. 결제 성공 후 premium 활성화 검증 방식을 정한다.
3. 월간 건강 리포트 PDF를 프리미엄 대표 기능으로 완성한다.

2. 티모+예원: 마이펫 데일리 대시보드 IA 정리
3. 로율+코다리: AI 결과 카드 고지 문구 및 탈퇴/데이터 삭제 흐름 검증
4. 영숙: 설정 탭 서비스 상태 패널 문구 정리
5. 현빈+데이브: 월간 리포트가 프리미엄 전환에 주는 가치 구조 정리

## 합의된 원칙

- 새 탭을 더 만들지 않는다.
- 첫 화면에서 사용자가 오늘 해야 할 일을 바로 보게 한다.
- AI/결제/동기화처럼 실패할 수 있는 기능은 상태를 숨기지 않는다.
- 데모 fallback은 유지하되 운영 모드와 명확히 구분한다.
- 건강 분석은 진단이 아니라 케어 참고 정보로 표현한다.
