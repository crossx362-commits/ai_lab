## 1. AI 호출 우선순위 (gemini_client.py)

모든 에이전트의 AI 호출은 아래 순서를 따른다:

| 순위 | 엔진 | 조건 | 비고 |
|------|------|------|------|
| 1 | Ollama 로컬 | `is_available()` == True | 무료, lm_first=True 시 |
| 2 | GPT-4o mini | `OPENAI_API_KEY` 환경변수 설정 시 | **기본값** (2026-06 변경) |
| 3 | Ollama 폴백 | GPT 실패 시 | 최종 폴백 |

**변경 이력 (2026-06-17):**
- Gemini API 할당량 지속 초과로 GPT-4o mini로 전환
- 폴백 체인: GPT-4o mini → Ollama (Gemini 제거)

```python
from _shared import gemini_client as _gc

result = _gc.text(prompt, task="blog")   # 텍스트 생성
img    = _gc.image(prompt)               # 이미지 생성
desc   = _gc.vision(img_bytes, prompt)   # 이미지→텍스트
```

**AI 사용 시 모델 표시 원칙 (필수):**
- 로컬 Ollama 사용 시: `[로컬 AI → {model_id}]` 자동 출력 (`ollama_client.chat()` 내장)
- GPT-4o mini 사용 시: `[GPT-4o mini] 텍스트 생성 완료` 출력
- 에이전트는 별도 구현 불필요 — `_gc.text()` / `lm_chat()` 호출 시 자동 표시됨

**task 라우팅:**
- `task="coding"` → Ollama DeepSeek 전용 (코딩 작업)
- `task="blog"` → Ollama Qwen 계열 모델 우선
- `task=""` → Ollama 자동 선택 (기타 에이전트)

**🆕 토큰 최적화 (2026-06-17 추가):**
- System instruction: 2KB 이하로 압축
- Prompt: 구분자 압축 (`BTC:95M|상승|RSI:45`)
- Output: max_tokens=500, temperature=0.1 (판단 작업)
- Few-shot 예시 2-3개 + Chain-of-Thought 필수
- Structured Output (Pydantic schema) 적용
- 상세: `공용스킬/AI_토큰_최적화_스킬.md` 참조

## 5. Ollama 모델 라우팅 및 자동 복구 (ollama_client.py)

```python
from _shared.ollama_client import chat, is_available
chat(prompt, task="coding")  # DeepSeek 전용
chat(prompt, task="blog")    # Qwen 전용
chat(prompt, task="")        # 자동 선택
```

### 모델 라우팅
- `task="coding"` / `task="blog"` — **strict 모드**: 해당 모델 없으면 `None` 반환 (다른 모델 낙하 없음)
- 모델 감지 결과 5분 캐싱

### 🔄 **자동 복구 로직 (모든 에이전트 공통 적용)**
`is_available()` 또는 `chat()` 호출 시 Ollama 연결 실패 감지 → **자동 복구**

1. **연결 실패 감지**: Ollama API 무응답 또는 모델 없음
2. **복구 스크립트 실행**: `_shared/ollama_client.py` 내장 복구 로직 자동 실행
3. **자동 재시도**: 최대 30회 (10초 간격) 대기하며 복구 완료 확인
4. **복구 완료**: Ollama 정상 복구 시 작업 재개
5. **복구 실패**: 30회 재시도 후에도 실패 시 `None` 반환 (텔레그램 알림 없음)

**중요**: 모든 에이전트는 `ollama_client`만 사용하며, 직접 ollama import 금지. 자동 복구는 `_shared/ollama_client.py` 내부에서 처리되므로 에이전트별 별도 구현 불필요.

## 6. 전 에이전트 공통 금지 규칙

1. **중복 데이터 취급 금지**: 최근 처리한 것과 동일·유사한 내용(유사도 ≥ 60%)은 재처리·재업로드하지 않는다.
2. **Drive URL 이미지 금지**: Google Drive 이미지 URL은 Blogger에서 렌더링 불가 — ImgBB/Catbox/Pollinations만 사용.
3. **API 키 하드코딩 금지**: 모든 인증 정보는 환경변수로만 관리.
4. **URL 직접 생성 금지**: 유튜브·외부 링크를 임의로 만들어 제공하지 않는다.

## 9. AI 에이전트 자동화 학습 결과 (2026-05-29 토론)

### Ollama 기반 멀티에이전트 핵심 원칙
1. **단계별 구현** — 에이전트 역할을 명확히 분리하고 작은 단위로 테스트

3. **Observable Pattern** — 비동기 에이전트 간 상태 변화를 이벤트로 발행·구독

## 8. 에이전트 공통 행동 프로토콜 (Governance)

### 정기 임무 (사전 승인 불필요, 자율 수행)
- 영숙: 텔레그램 응대·일정 관리 (상시)
- 소미: 정기 주식 보고 (오전/마감) + 15분 매수제안 슬롯
- 마켓데스크: 07:50 / 15:20 시장 종합
- 행크·유나·레온: 권역별 리서치 (미국/아시아/유럽)
- CEO(예원) 보고: 매일 오전 9시

### 신규 임무·예외 상황 — Dual-Gate 승인 적용
1. **비서(영숙)** 일정 산정 확인
2. **CEO(예원)** 최종 승인 확인

### 소통 규칙
- 모든 보고·질문·조율은 **비서 채널**을 통해 단일 창구로 처리
- CEO·타 에이전트에게 직접 보고 또는 직접 지시 요청 금지
- 업무 완료/실패 보고는 반드시 **비서 채널**로만 전달

### 텔레그램 알림
모든 에이전트 알림은 `_shared/telegram_notifier.py` 단일 모듈만 사용합니다.

### ⚡ 실시간 단계별 진행 보고 원칙 (전체 에이전트 공통)
모든 에이전트는 장시간 소요되는 파이프라인(예: 콘텐츠 생성, 렌더링, 정기 품질 스캔 등)을 구동할 때, 작업이 완전히 끝날 때까지 대기하지 않고 **각 단계의 진입 및 완료 시점에 실시간으로 단계 번호와 작업 내용을 텔레그램으로 전송**해야 합니다.
- **보고 의무 대상**: 2단계 이상의 멀티 스텝을 가진 모든 파이프라인 (백업/인프라 점검 포함).
- **보고 형식 예시**: `[에이전트명] N단계: [현재 상태 및 작업 대상]` 형태로 투명하게 알림을 발송합니다.
- **길이 제약**: 모바일 텔레그램의 가독성을 보장하기 위해, **각 알림 단계 보고 메시지는 줄바꿈을 포함하여 반드시 간단한 2~3줄 사이로만 작성**해야 합니다.
- **목적**: 작업의 불투명성을 방지하고, 에러나 중단이 발생한 단위를 사전에 투명하게 확인하여 디버깅 효율을 극대화합니다.

## 10. 프론트엔드 UI 템플릿 킷 (Shared UI Kits)

### [A] Developer UI Kits (React / TypeScript)
위치: `ai-team/assets/brain-seeds/40_템플릿/developer/`
고도로 모듈화된 4가지 프론트엔드 UI 킷이 준비되어 있다. 각 킷 내부의 `README.md` 및 `manifest.json`을 참조하여 컴포넌트를 조립할 것.

3. **`mobile-kit`**: 모바일 웹/앱 스크린 뷰 (`HomeScreen`, `ProfileScreen`, `SettingsScreen` 등)

## 12. 코딩 행동 지침 (CLAUDE.md 공통 적용)

모든 에이전트가 코드를 작성·수정할 때 반드시 준수한다.

### 12-1. 실행 전 사고
- 가정은 명시적으로 선언하고, 불확실하면 질문한다.
- 여러 해석이 가능하면 침묵하지 말고 선택지를 제시한다.
- 더 단순한 방법이 있으면 먼저 언급한다.

### 12-2. 단순성 우선
- 요청받은 것 이상의 기능을 추가하지 않는다.
- 단일 사용 코드에 추상화 레이어를 만들지 않는다.
- 200줄로 쓴 코드가 50줄로 가능하다면 다시 쓴다.

### 12-3. 수술적 변경
- 변경 요청과 직접 관련 없는 주변 코드는 손대지 않는다.
- 기존 스타일을 그대로 따른다 (개인 취향 반영 금지).
- 자신의 변경으로 생긴 미사용 import/변수만 정리한다.

### 12-4. 목표 기반 실행
- 작업을 검증 가능한 목표로 변환한다: "캡션 수정" → "50자 이상 캡션 생성 후 업로드 완료"
- 멀티스텝 작업은 계획을 먼저 출력하고 각 단계 완료 시 확인한다.
- 목표가 불명확하면 실행 전에 반드시 명확화한다.

### 12-5. 에이전트 공통 코드 규칙
- 주석은 WHY가 명확할 때만 작성 (WHAT 설명 주석 금지).
- 보안 취약점(XSS, SQL injection, command injection) 코드 절대 금지.
- 에러 핸들링은 실제 발생 가능한 케이스에만 추가한다.

## 11. 통합 에이전트 지식 베이스 (Knowledge Base)

모든 에이전트는 리서치, 웹 검색, 트렌드 분석 등 새로운 정보나 인사이트를 학습했을 경우, 반드시 이를 휘발시키지 않고 중앙 지식 베이스에 영구 저장해야 한다. 이렇게 저장된 지식은 추후 다른 에이전트가 `search_knowledge`를 통해 호출하여 자신의 컨텍스트로 활용할 수 있다.

```python
from _shared.knowledge_base import store_knowledge, search_knowledge

# 지식 저장 (리서치 종료 시 필수 호출)
store_knowledge(
    agent_name="마켓데스크", 
    topic="2026 하반기 시장 트렌드", 
    content="상세 분석 리포트 내용...", 
    tags=["시장", "트렌드"]
)

# 지식 검색 (기획 전 과거 자료 참조 시 호출)
past_insights = search_knowledge("SaaS 트렌드")
```
